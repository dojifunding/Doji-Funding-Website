Technical Specification Challenge Configurator

Complete reference for replicating the Doji Funding pricing configurator.
Covers architecture, state, pricing algorithm, UI components, and all interaction rules.

File: assets/js/configurator.js  ·  Config: config/pricing.php  ·  CSS: assets/css/configurator.css  ·  Version: 1.6 — April 2026

1. Architecture #arch

The configurator is a self-contained IIFE module (Immediately Invoked Function Expression). It exposes a minimal public API on window.Configurator. All internal state and helpers are private.

const Configurator = (function() {
    'use strict';
    // private state, helpers, builders…
    return { /* public API */ };
})();

// Auto-init when DOM ready
document.addEventListener('DOMContentLoaded', () => Configurator.init());

Data Flow

Pricing data originates in PHP and is injected into the page as a JSON global before configurator.js loads.

/* config/pricing.php */
getPricingJson()  // returns JSON: { accountSizes, basePrices, promoCodes }

/* In the page template (e.g. challenges.php) */
<script>
  window.DOJI_CONFIG = {
    pricing:   <?= getPricingJson() ?>,
    isLoggedIn: <?= $isLoggedIn ?>,
    presets:   <?= getPresetsJson() ?>,       // optional — competitor presets
    user:      { affiliateSales: <?= $sales ?> },
    affiliate: { unlockAt: 5 }
  };
</script>
<script src="assets/js/configurator.js"></script>
/* Inside configurator.js — reads the global */
const data          = window.DOJI_CONFIG?.pricing || {};
const accountSizes  = data.accountSizes  || [];
const basePricesData = data.basePrices   || {};
const promoCodes     = data.promoCodes   || {};

2. State Object #state

All configurator state lives in a single mutable object S. It is never replaced — only its properties are updated.

const S = {
    // Challenge type
    tab:         'onestep',   // 'onestep' | 'twostep'
    sizeIdx:     9,           // index into accountSizes[] — default = $50K (index 9)

    // Profit targets (one-step uses target; two-step uses target1 + target2)
    target:      10,          // one-step profit target %
    target1:     8,           // two-step phase 1 target %
    target2:     5,           // two-step phase 2 target %

    // Risk params
    daily:       5,           // daily loss limit %
    max:         10,          // max overall loss % — CONSTRAINT: daily ≤ max always

    // Funded account params
    split:       80,          // profit split %
    days:        5,           // min trading days (one-step default=5, two-step default=10)
    consistency: 30,          // consistency rule % (max single day vs total profit)

    // Loss calculation types
    dailyType:   'intraday',  // 'intraday' | 'eod' | 'static'
    maxType:     'intraday',  // 'intraday' | 'eod' | 'static'

    // Other toggles
    platform:    'dxtrade',   // 'dxtrade' (only option currently)
    payout:      'monthly',   // 'monthly' | 'biweekly' | 'weekly'
    overnight:   true,        // overnight holding add-on
    overweek:    false,       // overweek (weekend) holding add-on

    // Promo
    activePromo: null,        // { type:'percent'|'fixed', value:N, label:'' } or null
};

4. RATES Object #rates

All modifiers are stored as a fraction of the base price (bp). This makes the $ impact scale proportionally across all 19 account sizes. Calibration anchor: $50K one-step, bp = $249.

Formula: modifier $ = rate × bp  ·  Example: targetLow = 0.0803, bp = $249 → $20 per % below target default.

5. Pricing Algorithm #algorithm

calculatePrice() returns { basePrice, total, final, display }.

Full Formula
bp = basePricesData[tab][accountSizes[sizeIdx]] // lookup table
raw = bp + diff + spAdj + optAdj + ltAdj + eqSur + prSur + payAdj
total = max( round(raw), round(bp × 0.5) ) // price floor
final = max( round(total − discount), round(bp × 0.3) ) // promo floor
display = activePromo ? final : total

One Step — diff components

// Profit Target (anchor: 10%, free zone: 10–12%)
if (S.target < 10) diff += (10 - S.target) * r.targetLow  * bp;
if (S.target > 12) diff -= (S.target - 12) * r.targetHigh * bp;

// Daily Loss — two-tier surcharge (anchor: 5%)
if (S.daily > 5) {
    diff += Math.min(S.daily - 5, 2) * r.dailyHigh  * bp;  // +24.1% per % for 6–7%
    if (S.daily > 7) diff += (S.daily - 7) * r.dailyHeavy * bp;  // +40.2% per % for 8%+
}
if (S.daily < 4)  diff -= (4 - S.daily) * r.dailyLow   * bp;  // discount <4%

// Max Loss (anchor: 10%)
if (S.max > 10) diff += (S.max - 10) * r.maxHigh * bp;
if (S.max < 10) diff -= (10 - S.max) * r.maxLow  * bp;

// Min Days (anchor: 5)
if (S.days < 5) diff += (5 - S.days) * r.days * bp;

// Consistency (anchor: 40) — can be + or −
diff += (S.consistency - 40) * r.consistency * bp;

// Split (anchor: 70) — computed separately as spAdj
spAdj = (S.split - 70) * r.split * bp;

Two Step — diff components

// Combined target = target1 + target2 (anchor: 10–12%)
const tt = S.target1 + S.target2;
if (tt < 10) diff += (10 - tt) * r.targetLow  * bp;
if (tt > 12) diff -= (tt - 12) * r.targetHigh * bp;

// Daily Loss — same two-tier formula as one-step
// But discount threshold is 5 (not 4) for two-step
if (S.daily < 5) diff -= (5 - S.daily) * r.dailyLow * bp;

// Max Loss — same as one-step (anchor: 10%)

// Min Days (anchor: 10 for two-step)
if (S.days < 10) diff += (10 - S.days) * r.days * bp;

// Consistency (anchor: 45 for two-step)
diff += (S.consistency - 45) * r.consistency * bp;

// Split (anchor: 80 for two-step)
spAdj = (S.split - 80) * r.split * bp;

Common Adjustments (both tabs)

// Holding — overnight/overweek (both tabs use same keys)
if (!S.overnight && !S.overweek) optAdj -= r.overnight * bp;  // intraday-only discount
if (S.overweek)                  optAdj += r.overweek  * bp;  // weekend surcharge

// Loss type — applied to BOTH dailyType and maxType independently
if (S.dailyType === 'intraday') ltAdj -= r.ltIntraday * bp;
else if (S.dailyType === 'static')   ltAdj += r.ltStatic  * bp;
if (S.maxType   === 'intraday') ltAdj -= r.ltIntraday * bp;
else if (S.maxType   === 'static')   ltAdj += r.ltStatic  * bp;
// 'eod' type → no adjustment (neutral)

// Payout — only weekly adds cost; bi-weekly is included in base price
const payAdj = (S.payout === 'weekly' ? RATES.payWeekly : 0) * bp;

// Equal-loss surcharge (shown to user in warning box)
const eqSur = S.daily === S.max ? RATES.eqSurcharge * bp : 0;

// Pass-rate surcharge (silent — not shown to user)
const pr    = estimatePassRate();
const prSur = pr > 22 ? RATES.eqSurcharge * bp : 0;

6. Pass-Rate Estimation #passrate

estimatePassRate() returns a value clamped to 5–35%. It is a heuristic — not a real statistical model. Baseline: 8% (default config).

// ONE STEP
r = 8
r += max(0, 10 - S.target)       * 2.0   // easy target → more pass
r -= max(0, S.target - 10)       * 0.5
r += max(0, S.daily - 5)         * 1.5   // wide daily room
r += max(0, S.max - 10)          * 0.8   // wide max room
r -= max(0, 10 - S.max)          * 0.3
r += max(0, 5 - S.days)          * 0.8   // fewer min days = easier
r -= max(0, S.consistency - 30)  * 0.2   // strict consistency = harder

// TWO STEP
r = 8; tt = S.target1 + S.target2
r += max(0, 13 - tt)             * 1.5
r -= max(0, tt - 13)             * 0.4
r += max(0, S.daily - 5)         * 1.2
r += max(0, S.max - 10)          * 0.6
r -= max(0, 10 - S.max)          * 0.3
r += max(0, 10 - S.days)         * 0.4
r -= max(0, S.consistency - 35)  * 0.2

return Math.max(5, Math.min(35, r));
ZoneRangeEffect
SAFE< 15%No surcharge
WATCH15–20%No surcharge
DANGER20–22%No surcharge
CRITICAL> 22%+40.2% × bp surcharge (prSur) — silent, not shown to trader

Product decision: The pass-rate surcharge is included in the displayed price without being labelled separately. The warning zones are internal only and never rendered in the UI.

7. Price Floors #floors

Standard Floor

total = max(round(raw), round(bp × 0.5))

Prevents the calculated price from falling below 50% of base price, regardless of discounts from easy configurations.

Promo Floor

final = max(round(total − discount), round(bp × 0.3))

Prevents promo codes from discounting below 30% of base price.

8. Promo Code System #promo

Codes are case-insensitive on input (uppercased before lookup). Sourced from config/pricing.php → $promoCodes.

// Discount calculation

    ? total * S.activePromo.value / 100
    : S.activePromo.value;   // fixed $
final = Math.max(Math.round(total - disc), Math.round(bp * 0.3));

11. Event Handlers #events

onSliderLive(id, inputEl) — called on every drag tick (oninput)

Updates state immediately. Updates only the dragged slider's visual (fill width, thumb position, value label) via _updateSliderVisual(). Throttles the full updateUI() call via requestAnimationFrame — only one rAF frame queued at a time. Does not call buildSliders().

onSliderDone(id, val) — called on drag release (onchange)

Applies final constraints, then calls full buildSliders() + updateUI(). This is the "reconcile" pass that handles warning boxes, cascaded constraints, and hint divs.

onToggle(id, val)

Updates the corresponding S field. Special: when payout → 'weekly', clamps S.consistency to 30. Calls buildSliders() + updateUI().

onCheck(id, checked)

Handles overnight/overweek switches. Enforces the overnight↔overweek coupling rule. Calls buildSliders() + updateUI().
